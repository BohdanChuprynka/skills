# Configuration Reference

Everything `calendar-plan.sh` reads at runtime — CLI flags, env vars, config files — plus common recipes for the things you'll actually change.

> Resolution order (highest priority first): **CLI flag > env var > `config/settings.conf` > built-in default**

---

## CLI flags

| Flag | Type | Default | Description |
|---|---|---|---|
| `--date YYYY-MM-DD` | date | tomorrow in `TIMEZONE` | Target date to plan. Default resolves at run-time using `TIMEZONE`. |
| `--mode auto\|draft` | string | `DEFAULT_MODE` (typ. `auto`) | `auto` writes safe blocks to Google Calendar; `draft` reports only. |
| `--model ID` | string | `MODEL` from settings.conf or `claude-sonnet-4-6` | Anthropic model. Use `claude-opus-4-7` for complex weeks. |
| `--dry-run` | flag | off | Render the prompt + print the command — no LLM call. Sanity check before live runs. |
| `--help`, `-h` | flag | — | Show usage and exit. |

Examples:

```bash
./calendar-plan.sh                              # default: tomorrow, auto mode
./calendar-plan.sh --date 2026-05-19 --mode draft
./calendar-plan.sh --model claude-opus-4-7      # heavier model for one run
./calendar-plan.sh --dry-run                    # validate without calling Claude
```

## Environment variables

All are optional. Useful when running under launchd, cron, CI, or any context where shell args are awkward.

| Variable | Type | Default | Description |
|---|---|---|---|
| `MODEL` | string | `claude-sonnet-4-6` | Anthropic model. CLI `--model` wins. |
| `TIMEZONE` | IANA tz | `America/New_York` | Used for target-date resolution + launchd plist `TZ`. |
| `CRON_HOUR` | int 0-23 | `22` | The hour the scheduled run is expected to fire. Influences the prompt's "Target date" rule. |
| `DEFAULT_MODE` | `auto`/`draft` | `auto` | Mode used when `--mode` is not passed. |
| `CALENDAR_CONTEXT` | path | (none) | Absolute path to your Calendar Context markdown. Read each run. |
| `TASK_SOURCE_NAME` | string | (none) | Title of the Notion page that holds the daily task sequence. |
| `EXTRA_ADD_DIRS` | space-sep paths | (none) | Extra `--add-dir` paths passed to `claude`. Useful when the planner needs to read other vault pages. |

These are loaded from `config/settings.conf` (which `setup.sh` writes). Direct shell export overrides the file:

```bash
MODEL=claude-opus-4-7 ./calendar-plan.sh
TIMEZONE=Europe/Lisbon ./calendar-plan.sh
```

## Config files

Three files in the skill's `config/` directory drive behavior. They ship as `.example.*` templates; copy to the non-example name to activate.

### `config/settings.conf`

Bash key=value file sourced by `calendar-plan.sh`. Holds the env vars listed above.

```bash
# Generated by setup.sh — chmod 600
MODEL="claude-sonnet-4-6"
TIMEZONE="America/New_York"
CRON_HOUR="22"
CALENDAR_CONTEXT="/Users/<you>/Documents/Obsidian/me/wiki/Calendar Context.md"
TASK_SOURCE_NAME="12-Week Planner"
DEFAULT_MODE="auto"
EXTRA_ADD_DIRS=""
```

Sourced as bash — quote string values. Never store tokens here (use `mcp-config.json` instead; bash escaping is poor).

### `config/planning-preferences.md`

User-specific, free-form markdown the planner reads as part of its prompt. Holds the load-bearing personalization:

- Calendar IDs for each sub-calendar (`School`, `Personal`, `Work`, primary)
- Notion task-source page title + fallback UUID
- Daily defaults (wake times, deep-work golden hours, workout windows)
- Commute assumptions
- Calendar routing (which sub-calendar each block type goes on)
- Audit-derived corrections (rules learned from past runs)

Copy from `examples/planning-preferences.example.md` and edit. The example file has placeholders like `<Calendar Label 1>` — replace with real values.

### `config/mcp-config.json`

Defines MCP servers active during runs. **Not auto-created** — only generated by `setup.sh --mcp` or hand-edited.

See [`MCP-SETUP.md`](MCP-SETUP.md) for the full schema and per-MCP setup walkthroughs. See [`config/mcp-config.example.json`](../skills/calendar-plan/config/mcp-config.example.json) for a starter template.

This file is gitignored. **Do not commit it.** It contains tokens.

## Common recipes

### "I want to plan a specific date, not tomorrow"

```bash
./calendar-plan.sh --date 2026-05-19
```

### "I want to use Opus for one run"

```bash
./calendar-plan.sh --model claude-opus-4-7
```

Or permanently for all runs:

```bash
# edit config/settings.conf
MODEL="claude-opus-4-7"
```

### "I want to run draft mode tonight, not write to calendar"

```bash
./calendar-plan.sh --mode draft
```

Or make it the cron default:

```bash
# config/settings.conf
DEFAULT_MODE="draft"
```

### "I'm experimenting with prompts and don't want to spend money"

```bash
./calendar-plan.sh --dry-run
```

Renders the prompt + prints the command. No LLM call.

### "I moved to a different timezone"

```bash
# config/settings.conf
TIMEZONE="Europe/Lisbon"
```

Also update the launchd plist's `TZ` env var if you're using the cron — otherwise launchd fires at the wrong wall time.

### "I want the cron to fire at a different time"

```bash
# config/settings.conf
CRON_HOUR="23"
```

Also update the launchd plist's `StartCalendarInterval.Hour`. The plist is the actual scheduler; `CRON_HOUR` in settings.conf only affects the wording in the prompt ("This cron runs at X:00").

### "I want the planner to read additional vault pages"

```bash
# config/settings.conf
EXTRA_ADD_DIRS="/Users/<you>/Documents/Obsidian/projects /Users/<you>/Documents/Obsidian/learning"
```

Space-separated absolute paths. Each becomes a `--add-dir` arg. Use for cross-referencing project deadlines or learning notes that should affect block intensity.

### "My Notion page is named differently"

```bash
# config/settings.conf
TASK_SOURCE_NAME="Daily Plan 2026"
```

Also update `config/planning-preferences.md` → "Notion: private page titled `<your title>`" + the fallback page UUID.

### "I rotated my Notion token"

```bash
./setup.sh --mcp
```

Re-runs ONLY the MCP step. Updates `config/mcp-config.json`.

Or edit directly:

```bash
$EDITOR config/mcp-config.json
# replace the ntn_xxx value in OPENAPI_MCP_HEADERS
```

### "I want to disable Gmail signal without uninstalling the MCP"

Comment out the gmail entry in `config/mcp-config.json` (note: JSON doesn't support comments — delete the entry, or wrap it in an unused key like `"_disabled_gmail": {...}`).

The planner will note `gmail: missing-tool` in memory and continue with the remaining sources.

### "I want to run it manually right now, not wait for the cron"

```bash
# launchd target:
launchctl start com.user.calendar-plan

# or direct:
~/.claude/skills/calendar-plan/calendar-plan.sh --mode auto

# Codex target:
codex automations run calendar-plan
```

### "I keep tokens at non-default paths"

Edit `config/mcp-config.json`. The `env` blocks for each MCP define where credentials live. Absolute paths only (JSON doesn't expand `~`).

```json
"google-calendar": {
  "env": { "GOOGLE_OAUTH_CREDENTIALS": "/opt/secrets/gcal/creds.json" }
}
```

### "I want a different `claude` binary for testing"

```bash
PATH="/path/to/test-claude:$PATH" ./calendar-plan.sh --dry-run
```

### "I want to inspect what the planner would see without running it"

```bash
./calendar-plan.sh --dry-run
cat /tmp/calendar-plan-prompt.*.md
```

Shows the rendered prompt with all placeholders resolved.

## Codex target configuration

The Codex target reads its config from a different place — Codex's own automation system at `~/.codex/automations/calendar-plan/automation.toml`. Edit that file directly to change:

- `model` (e.g. `gpt-5.5`)
- `reasoning_effort` (e.g. `xhigh`)
- `rrule` (e.g. `RRULE:FREQ=DAILY;BYHOUR=22;BYMINUTE=0`)
- `cwds` (working directory for the run)
- `prompt` (the long body — note: editing here desyncs from `prompts/cron-prompt.md`. Prefer re-running `bash codex/setup.sh`)

The prompt single source of truth is `prompts/cron-prompt.md`. Re-running `codex/setup.sh` re-renders `automation.toml` from that file with placeholders resolved, keeping the two runtimes in sync.
